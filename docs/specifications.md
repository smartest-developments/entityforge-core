# Specifications

## Scope

This repository is primarily a Senzing evaluation and reporting pipeline centered on `core/`.

The active flow is:

1. ingest source JSON or JSONL
2. map records into Senzing-ready JSONL
3. create and load a Senzing project
4. export comparison artifacts
5. build per-run dashboard bundles
6. generate Senzing audit outputs
7. generate per-run non-match analysis bundles

## Canonical entrypoints

- `core/run_production_command.sh`
- `core/app/run_mvp_pipeline.py`
- `core/app/run_mvp_with_auto_diagnosis.py`
- `core/app/prepare_senzing_audit_inputs.py`
- `core/app/build_non_match_why_report.py`

## Canonical runtime layout

Run outputs belong under the configured runtime root:

```text
<RUNTIME_DIR>/runs/<run_id>/output_bundle/
```

Observed defaults:

- production wrapper default: `/mnt/runtime`
- local smoke examples committed in the repo: `core/runtime_smoke*`, `core/runtime_demo_*`

Expected subfolders:

- `technical output/`
- `dashboard_web/`
- `dashboard_streamlit_app/`
- `senzing_audit/`
- `non_match_why/`
- `diagnostics/` when generated
- `limited_inputs/` when `INPUT_RECORD_LIMIT` is used

## Dataset behavior

The repository supports both:

- synthetic stress datasets generated by `core/app/generate_sample_inputs.py`
- production-style input files such as `Senzing-Ready.json`

Known documented dataset semantics:

- `SOURCE_TRUE_GROUP_ID` is used for synthetic truth
- `IPG ID` or `SOURCE_IPG_ID` is used as clustering baseline for audit and comparison
- `RECORD_ID` is generated by the mapper when missing from source input

## Output expectations

For a successful production run, the repository should be able to produce:

- mapped JSONL
- run summary
- management summary
- ground truth quality report
- dashboard web bundle
- dashboard streamlit bundle
- Senzing audit package
- non-match WHY analysis bundle

## Assumptions

- `core/` is the active source of truth
- `S3Z/` contains reusable and partially legacy material
- some older docs still reference historical paths such as `output/` or `mvp_runtime`; treat those as legacy unless current `core/*.sh` or `core/app/*.py` still use them
